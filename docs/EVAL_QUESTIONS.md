# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문



## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.



## 프로젝트 전체

- 이 서비스가 해결하는 문제는 무엇인가?
취업 준비를 하는 사람이 자신의 이력서나 조건에 맞는 복잡한 채용 공고를 일일이 찾고 분석하는 시간과 번거로움을 줄여준다.

- RAG를 사용하지 않으면 어떤 문제가 생기는가?
LLM이 최신 채용 공고 데이터를 파악하기 어려워 과거 정보로 답변하거나 환각을 지어낸다.

- 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?
프론트엔드 배포 과정에서 CORS 도메인 격리 및 도커 빌드 시점의 환경 변수가 꼬이는 문제가 가장 어려웠다. 각 파일에 지정된 localhost를 확인하고 로컬과 배포된 사이트 각각에서 필요한 백엔드를 참고할 수 있도록 환경변수 코드를 수정하였다.



## 데이터 관련

- 왜 이 데이터를 선택했는가?
구직자의 역량 매칭 및 피드백을 정확하게 수행하기 위해서 선택했다.

- 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?
목업 데이터로 주로 구성되어 있으며 공고 마감일이 지났거나 각종 자격 요건이 변경되었을 수도 있다.

- SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?
SQLite는 공고 내용의 원본과 상세 텍스트 조회를 담당하고 ChromaDB는 의미 기반의 고속 벡터 유사도 검색을 담당한다.

- 결측치와 중복을 어떻게 처리했는가?
공고 ID 기반으로 중복 항목을 사전에 제거하고 필수 자격 요건이 비어있는 결측치는 수집 단계에서 기본 문구로 대체하거나 드롭했다.



## RAG 관련

- RAG 구조에서 ChromaDB는 어떤 역할을 하는가?
사용자의 질문(이력서)을 벡터로 변환해 가장 연관성이 높은 채용 공고 상위 N개를 빠르게 찾아내는 검색 엔진 역할을 한다.

- n_results 값을 높이면 어떤 트레이드오프가 생기는가?
더 많은 공고 정보를 LLM에 넘겨주어 분석의 다양성은 늘어나지만 LLM의 토큰 비용이 증가하고 답변 속도가 느려진다.

- 검색 결과 품질을 어떻게 검증했는가?
특정 기술 스택 쿼리를 던졌을 때 실제 해당 기술을 요구하는 채용 공고가 상위 매칭되는지 키워드와 의미적 일치도를 수동으로 검증했다.

- Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?
LLM이 실제로 없는 자료를 만들어 내면서 그럴듯한 거짓말을 하는 현상이며 RAG는 데이터베이스에서 찾은 확실한 실제 공고 문서를 LLM에 근거로 제공하여 이를 줄인다.

## API 관련

- /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?
사용자의 이력서나 자기소개서 같은 대용량 텍스트 데이터를 URL 길이 제한 없이 안전하게 본문에 담아 전달하기 위해서이다.

- 응답 JSON에 sources를 포함시킨 이유는 무엇인가?
LLM이 어떤 채용 공고를 바탕으로 추천 결과를 도출했는지 사용자가 원본 공고의 출처를 직접 눈으로 검증할 수 있게 하기 위함이다.

- /health 엔드포인트가 왜 필요한가?
Render 클라우드 서버나 로컬 인프라가 현재 작동 중인지 실시간으로 모니터링하고 배포 상태를 체크하기 위해 필요하다.


## 보안 관련

- .env 파일을 GitHub에 올리면 어떤 일이 생기는가?
기밀 인증 정보가 공유되어 타인에게 무단 도용당하거나 막대한 비용 문제를 겪을 수 있다.

- .env와 .env.example의 차이는 무엇인가?
.env는 실제 비밀키가 담긴 개인 보안 파일리고, .env.example은 비밀키 없이 어떤 키값들이 필요한지 규격만 보여주는 공유용 템플릿이다.



## Docker 관련

- Docker를 사용하는 이유는 무엇인가?
개발 환경 그대로를 캡슐화하여 어디서나 똑같이 실행하기 위함이다.

- Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?
EXPOSE는 컨테이너 내부에서 외부로 개방할 포트 번호를 명시하고, CMD는 컨테이너가 켜지는 순간 실행할 최종 명령어를 저장한다.
